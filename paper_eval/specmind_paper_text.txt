Abstract
Specifications are vital for ensuring program
correctness, yet writing them manually re-
mains challenging and time-intensive. Re-
cent large language model (LLM)-based meth-
ods have shown successes in generating spec-
ifications such as postconditions, but exist-
ing single-pass prompting often yields inaccu-
rate results. In this paper, we present SPEC-
MIND, a novel framework for postcondition
generation that treats LLMs as interactive
and exploratory reasoners rather than one-shot
generators. SPECMINDemploys feedback-
driven multi-turn prompting approaches, en-
abling the model to iteratively refine candi-
date postconditions by incorporating implicit
and explicit correctness feedback, while au-
tonomously deciding when to stop. This pro-
cess fosters deeper code comprehension and im-
proves alignment with true program behavior
via exploratory attempts. Our empirical evalu-
ation shows that SPECMINDsignificantly out-
performs state-of-the-art approaches in both ac-
curacy and completeness of generated postcon-
ditions. Our data and code are available here
https://anonymous.4open.science/r/SpecMind
1 Introduction
Program specifications (pre-/postconditions) are
central to checking that program behavior matches
intent, but writing them manually is tedious. Prior
work on automated inference broadly falls into four
categories. First, program-analysis methods infer
invariants dynamically from executions (Beschast-
nikh et al., 2011; Ernst et al., 1999)—and thus are
coverage-limited—or statically from code, often
at the cost of conservative, false-positive–prone re-
sults (Ramanathan et al., 2007; Wei et al., 2011).
Second, data-mining approaches extract common
API-usage patterns (e.g., call pairs/sequences and
automata) from large codebases (Wasylkowski
et al., 2007; Thummalapenta and Xie, 2009; Pradel
and Gross, 2009; Wasylkowski and Zeller, 2009;
Zhong et al., 2009a), but typically do not infer se-
mantic specifications such as pre-/postconditions.
Third, EvoSpex (Molina et al., 2021) uses
execution-guided evolutionary search to evolve
postcondition candidates, yet its handcrafted opera-
tors only weakly exploit program semantics, lead-
ing to incomplete or brittle postconditions. Fourth,
nl2postcond (Endres et al., 2024) leverages LLMs
to translate code and informal documentation into
postconditions. The limitation of simple prompting
in nl2postcond lies in their reliance onsingle-pass
generation: nl2postcond expects the LLM to gen-
erate a correct and complete postcondition in a
single prompt. Such prompting results inincorrect
postconditionsthat are syntactically plausible but
semantically inaccurate, failing correctness tests
or lacking discriminative power for bug detection.
(Related work details are given in Appendix A.1).
In our work, SPECMIND, we aim to enhance
the LLM’s capability to generate postconditions
from given source code. A more robust alternative
isretry-based prompting(Greedy Multi-turnor
Greedy), where the LLM iteratively regenerates
postconditions until one passes a soundness check
and exceeds a completeness threshold. While this
introduces feedback, it remainsreactive and exter-
nally controlled—the LLM is not actively involved
in reasoning but simply retries in response to fail-
ure signals. Thus, we proposeExploratory Multi-
Turn(Exploratory), a more cognitively aligned
framework where the LLM is treated not just as
acode-to-code generator, but as aninteractive
reasoner. We prompt the LLM to autonomously
explore, internalize the reasoning process to derive
the postcondition, and decide when to stop.We
prompt the model to perform one or multiple ex-
ploration turns, i.e., submitting postconditions to
receive intermediate feedback on soundness and
completeness before submitting the final postcondi-
tion.This method engages the model in an iterative
self-refinement process where it internalizes the

2 Illustration
The function search in Fig. 1takes a non-empty list
of positive integers as input and returns the great-
est integer in the list whose frequency is greater
than or equal to its value; if no such integer exists,
it returns −1. In the ground truth, the complete
specification for this function must contain all of
the following sub-conditions:
(1) the return value is either −1 or a positive
integer from the input list;
(2) if the return value is positive, its frequency
must be greater than or equal to its value;
(3) if the return value is positive, it must be the
greatest integer satisfying the frequency condition;
(4) if the return value is −1, no integer in the list
satisfies the frequency condition.
We applied the state-of-the-art, LLM-based,
postcondition generation tool, nl2postcond (Endres
et al., 2024) on the code of the search function.
nl2postcond leverages asingle-passprompting strat-
egy to request Llama 4 Scout to generate the post-
condition for the given code.
A candidate postcondition can be evaluated us-
ing two metrics:correctness and completeness(En-
dres et al., 2024).Correctnessrefers to if the post-
condition passes all test cases in the given test set.
A postcondition is correct if it holds true for all
test cases that the original function passes (no con-
tradiction to the behavior in test cases). However,
correctness alone is not sufficient as a trivial post-
condition like "True" is always correct but useless

in bug detection.Completeness, referring to dis-
criminative power, measures how informative the
postcondition is. It reflects the postcondition’s abil-
ity to detect incorrect versions of the function (e.g.,
mutated code). This is done by checking if the
postcondition can "catch" mutants (i.e., detecting
bugs), i.e., the modified versions that behave dif-
ferently. Formally, completeness is the percentage
of mutants that can be distinguished (i.e., rejected)
by the postcondition when used with the test suite.
We made the following observations:
1. The state-of-the-art LLM-based, Single-pass
approach, nl2postcond (Endres et al., 2024), gen-
erated the postconditions ensuring that the return
value is either -1 or a positive integer from the input
list, thereby satisfying the condition (1). However,
none of the conditions (2)–(4) was generated.
2. Feedback-driven Multi-Turn Approaches.
Prior work suggests that the feedback helps LLMs
ground their generations more effectively (Gehring
et al., 2025). Building on this, we integrated
feedback into the process by explicitly indicating
which generated constraints were incorrect or non-
discriminative. Thus, we propose the following
two feedback-driven Multi-Turn approaches:
Greedy Multi-Turn.The first approach em-
ployed a simple feedback mechanism in which
Llama 4 Scoutrepeatedly attempted to produce
a correct postconditionuntil all test cases passed
and a desired completeness is reached, using feed-
back from previous failed attempts. This feedback
includes either binary pass/fail signals or syntax
errors. The iterative process enabled the model to
progressively cover the condition (1) in the first
attempt, the condition (2) in the second, and the
condition (3) in the third, eventually combining
them into a single assertion by the fourth attempt.
Within the 12-attempt limit, the model focused pre-
dominantly on these three conditions, but failed to
generate the condition (4). The generated asser-
tions consistently began with assert return_value
== -1 or ..., a pattern that could be trivially satis-
fied by faulty code returning−1in all cases.
Exploratory Multi-turn.In real-world learning,
students rarely solve complex problems correctly
in a single attempt. Effective learners instead ask
clarifying questions, explore partial solutions, and
test their understanding before committing to a fi-
nal answer. This differs from blindly retrying after
a “try again” signal, and from single-pass solving
without refinement. Inspired by this, we propose
Exploratory Multi-turn for postcondition genera-
tion, where the LLM acts not as a code-to-code gen-
erator but as anactive reasoner. It autonomously
explores intermediate reasoning paths, probes par-
tial answers, internalizes feedback, and decides
when it has enough evidence to produce the result.
In the example, similar to Greedy Multi-turn,
the first three attempts correctly generated condi-
tions (1)–(3). In the 4th attempt, a minor change
triggered an AssertionError. Unlike Greedy, which
often stagnates on the same incomplete assertion,
Exploratory Multi-Turn broke from its prior pattern
and shifted direction in the fifth attempt. The model
explicitly noted:“The assertion failed, which
means our current postconditions might not be suffi-
cient or accurate. Let’s revisit and refine them... We
can use a different approach... Let’s iterate over... ”.
This shift produced a postcondition that covered
condition (4) and discarded the overly permissive
pattern assert return_value == -1 or .... It finally
generated the correct and complete postcondition,
suggesting that Exploratory Multi-Turn helps es-
cape local optima and improves completeness.
3 Problem Formulation
Given a function f:I → O , where I and O are the
input and output domains, the goal is to synthesize
a postcondition ϕ:I × O → {true,false} that
characterizes the correct behavior off.
The postcondition ϕ is a logical predicate that
must hold for any input-output pair (i, o) where
o=f(i) . Formally: ∀i∈ I, ϕ(i, f(i)) =true.
In practice, ϕ is often expressed as an assertion or
a condition over program variables after execution.
The goal is to infer such a ϕ automatically, using
the following inputs: a function’s source code, a
finite test suite T={(i 1, o1), . . . ,(in, on)}, and a
set of mutants M of the original f. Two evaluation
criteria for a synthesized postconditionϕinclude
Correctness (Corr.): The postcondition must
hold on all known correct executions captured in
the test suite: ∀(ij, oj)∈ T, ϕ(i j, oj) =true.
The postcondition must reject incorrect variants.
Completeness (Comp.): Given a set of program
mutants M={f ′
1, f ′
2, . . . , f′
m}, the completeness
of a postcondition measures the proportion of mu-
tants for which the postcondition fails:
Comp(ϕ) = |{f ′
k ∈ M | ∃i∈ I, ϕ(i, f ′
k(i)) =false}|
|M|
Correctness ensures consistency with the given
code, while completeness reflects the specifica-
tion’s robustness in identifying deviations/defects.

5 Prompt Design and Feedback
Prompt Design. We adopt a modular and evolv-
ing prompt template that reflects the model’s cur-
rent understanding, errors in prior attempts, and
the functional structure of the code under analysis.
The base prompt includes the full code of the tar-

get function and, optionally, a high-level natural
language description (if available). Moreover, each
subsequent prompt incorporates an interaction his-
tory consisting of prior postconditions, associated
feedback, and explicit instructions for refinement.
Formally, let the function be denoted f, and let the
interaction history up to turntbe represented as:
Ht = [(ϕ 1,feedback 1),(ϕ 2,feedback 2), . . . ,
(ϕt−1,feedback t−1)]
The prompt at turn t, denoted Promptt, is synthe-
sized as: Promptt =PromptTemplate(f,H t) where
PromptTemplate is a function that formats code, prior
generations, and critiques into an LLM-readable
input. A prompt template is shown in Fig. 3.
Our evolving structure allows the LLM to inter-
nalize past mistakes and progressively refine its
output. Unlike static prompt, our dynamic history-
aware formulation fosters deeper reasoning and re-
moves the need to reset the context between turns.
Feedback Mechanism. Our feedback mecha-
nism serves two goals: 1) guiding the LLM’s refine-
ment with actionable signals, and 2) providing a
quantitative estimate of each candidate’s semantic
completeness. Each LLM-generated postcondition
ϕt is evaluated against the code’s observed behav-
ior, typically using a test suite and a set of mutants.
The feedback feedbackt includes: (a) acorrect-
ness indicator–whether ϕt holds on all tests; and
(b) acompleteness score–the fraction of mutants
“caught” by ϕt (i.e., faulty variants that violate the
postcondition). If the completeness threshold is not
reached within the attempt budget, we also return
the remaining uncaught mutants as additional feed-
back. We append the prompt–feedback pair to the
history Ht to steer subsequent generations toward
remaining behavioral gaps. If available, we can use
symbolic execution or verification backends; oth-
erwise, correctness is checked empirically on tests
and completeness is measured via mutant catching.
Grounding prompts in correctness and complete-
ness yields an informative refinement loop.
6 Empirical Evaluation
We aim to answer the following questions:
RQ1.[Effectiveness–Efficiency]. How effective
and efficient is our tool in postcondition inference?
RQ2.[Reasoning]. How does SPECMINDwith
Exploratory Multi-turn perform reasoning?
RQ3.[Stratifying Results]. How does SPEC-

RQ4.[Cost Efficiency]. How efficient is SPEC-

Benchmarks. We selected EvalPlus as our dataset
as it was used in nl2postcond (Endres et al., 2024).
It has 164 Python problems, each with a function
stub, textual description, reference implementation,
and validation tests (Liu et al., 2023). EvalPlus
updates the HumanEval benchmark (Chen et al.,
2021), containing the same problems but with more
extensive test suites (≈ 775 test cases per problem).
We also used FixEval (Anjum Haque et al., 2023)
for bug detection experiment (see Appendix A.2).
Baselines. We chose as baseline the state-of-the-
art nl2postcond, with access to the reference imple-
mentation. nl2postcond was shown to outperform
prior approaches such as TOGA (Dinella et al.,
2022) and Daikon (Ernst et al., 1999). We used
Llama 4 Scout as the underlying LLM for both
nl2postcondand our tool in all experiments.

Limitations & Threats to Validity
Our evaluation is conducted on EvalPlus (Endres
et al., 2024), which may not reflect the full diver-
sity of real-world codebases. While EvalPlus is
a large public benchmark with postcondition an-
notations and enables a fair comparison against
nl2postcond on its original setting, it primarily con-
sists of Python tasks of moderate complexity. Thus,
generalization to larger systems, other program-
ming languages, or other forms of specifications
is not guaranteed. In addition, we evaluate mainly
GPT-family models; performance may differ for
other model families or future versions.
Our results may also be influenced by the under-
lying pre-training knowledge of LLMs. Although
we cannot fully rule out exposure to similar code
during pretraining, our evaluation centers on gener-
ating new postconditions and validating them via
execution, which is unlikely to be explained solely
by memorization.
We operationalize postcondition quality using
two proxies: correctness (passing the test suite) and
completeness (rejecting mutants). Both measures
inherit limitations from their backends: correctness
is bounded by test coverage, and completeness de-
pends on the quality and representativeness of the
mutant set. If the tests or mutants fail to capture
realistic behaviors or defects, our metrics may over-
estimate true specification quality or bug-detection
usefulness.
Finally, our conclusions depend on experimental
choices such as the attempt budget µ and com-
pleteness threshold τ. While we evaluate across
multiple configurations to reduce sensitivity to any
single setting, other datasets, feedback designs, or
hyperparameter choices could change the observed
trade-offs between effectiveness and efficiency.

A.1 Related Work
Researchers have proposed automated approaches
that fall into three categories:program analysis-
based,data mining-based, andlarge language
model (LLM)-based.
Program analysis-based methods use dynamic
or static techniques. Dynamic approaches (Am-
mons et al., 2002; Beschastnikh et al., 2011; Ernst
et al., 1999) infer properties by monitoring execu-
tions but depend heavily on test coverage. Static
analyses (Shoham et al., 2007; Engler et al., 2001;
Kremenek et al., 2006; Ramanathan et al., 2007;
Wei et al., 2011) and abstract interpretation (Cousot
et al., 2012) avoid this dependency but often pro-
duce conservative or imprecise specifications due
to high false-positive rates. Several approaches at-
tempted to generate specification by analyzing API
documentation or code comments (Pandita et al.,
2012; Tan et al., 2007, 2011, 2012; Blasi et al.,
2018; Zhong et al., 2009b; Zhou et al., 2017).
Data mining-based approaches extract common
API usage patterns from large codebases (Li and
Zhou, 2005; Livshits and Zimmermann, 2005;
Zhong et al., 2009a). They identify call pairs,
sequences, and finite-state models (Wasylkowski
et al., 2007; Williams and Hollingsworth, 2005;
Thummalapenta and Xie, 2009; Nguyen et al.,
2009; Pradel and Gross, 2009; Wasylkowski and
Zeller, 2009). While useful for API usages, they
rarely infer semantic specifications such as pre/-
postconditions (Ramanathan et al., 2007).
In general, ML approaches for specification
generation have advanced in multiple directions,
including postcondition inference (Endres et al.,
2024), synthesizing test oracles (Dinella et al.,
2022; Mastropaolo et al., 2023; Tufano et al., 2022),
improving test coverage (Lemieux et al., 2023; Alt-
mayer Pizzorno and Berger, 2025), and generating
unit tests (Lahiri et al., 2023; Tufano et al., 2021).

These techniques rely on different types of inputs.
AthenaTest (Tufano et al., 2021) generates both the
input and the oracle of a unit test directly from the
implementation of the focal method, while TOGA
generates only the oracle (Dinella et al., 2022).
TiCoder (Lahiri et al., 2023) leverages LLMs to
produce both inputs and outputs based on a natural
language description of user intent.
While these methods target the production of
concrete test cases (and sometimes oracles), our
work instead focuses on generating postcondi-
tions. Similarly, EvoSpex (Molina et al., 2021)
applies evolutionary learning to infer functional in-
put–output relationships. More recent efforts have
explored property-based specifications. Vikram
et al.(Vikram et al., 2024) propose leveraging
LLMs to generate property-based tests (PBTs),
while Speculyzer(Key et al., 2022) enumerates
likely properties and candidate inputs with LLMs,
but uses them primarily as heuristics to guide code
generation rather than to ensure correctness or com-
pleteness of specifications. Beyond input–output
properties, machine learning has also been applied
to infer intermediate specifications, such as pro-
gram invariants (Garg et al., 2016; Laich et al.,
2020; Yao et al., 2020; Pei et al., 2023).
A.2 Bug Detection via Generated
Postconditions
Experimental Methodology. In this experiment,
we aim to evaluate the usefulness of the gener-
ated postconditions. We adapted an evaluation ap-
proach used innl2postcond (Endres et al., 2024) and
TOGA (Dinella et al., 2022). The evaluation idea
is that it aims to evaluate if a model can generate
postconditions that distinguish between correct and
buggy code regarding regression and triggering test
cases in regression testing. We used FixEval (An-
jum Haque et al., 2023), a benchmark comprising
buggy and correct submission code to competitive
programming problems. From the full test set of
243k Python buggy-correct pairs, we filtered out
runtime errors and kept 157,355 pairs. We then
randomly selected 384 bugs, ensuring 95% statisti-
cal significance. We evaluated the effectiveness of
postconditions that a user could have used to catch
a bug as the postconditions fail on the buggy ver-
sion, and succeed on the correct one. The rationale
is as follows. Using the associated test suite for
a problem in the dataset to ensure correctness, as
we run a model on the correct version, the result-
ing postcondition ϕ acts as an oracle of intended
behavior of the code. When we apply that ora-
cle to the executions of the buggy solution, any
deviation from the intended relation between in-
puts and outputs will violate the postcondition ϕ
(i.e., there exists an input i such that postcondition
ϕ(i,buggy_function(i))=false). This is the same
completeness definition: postconditions “reject” a
faulty variant (mutant); here, the buggy version
plays the role of a single mutant. Thus, the gener-
ated postcondition can be used in detecting future
bugs in regression testing. We used the same crite-
ria in nl2postcond for abug-discriminatingpostcon-
dition. We consider a generated postcondition to be
bug-discriminating if it satisfies the following: (1)
The postcondition passes all the trigger and regres-
sion tests, on the correct version of a function. (2)
The postcondition fails a trigger test or a regression
test on the buggy version (Endres et al., 2024).
Experimental Results. As seen in Table 3, both
Greedy and Exploratory approaches are able to
distinguish more buggy versions than the baseline
nl2postcond being run µ times independently (for a
fair comparison). Whenµ=3 submission attempts,
the baseline distinguished 16.15% of the buggy ver-
sions from the correct ones, with passing 66.41%
of test cases. In contrast, Greedy distinguishes
relatively 53.19% more buggy versions with pass-
ing 59.38% of test cases. Importantly, Exploratory
catches2Xmore buggy versions than the baseline
with34.64%of passing test cases.
As we increased the number of submission at-
tempts to µ= 5 , all approaches performs better
in terms of both correctness and number of dis-
tinguishable bugs. As µ= 10 , Exploratory per-
forms the best as it distinguished39.58%of the
buggy versions, i.e., better than both the baseline
and Greedy.

with its best-performing one appearing in attempt

non-negative integer, and (2) it should be bounded

relevance. Greedy, however, reverts to the first
property and briefly mentions the case n≤2 in its
reasoning but dismisses it as irrelevant, despite its
semantic importance.
